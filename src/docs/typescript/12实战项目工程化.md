# 实战：项目工程化

> 最后一章，我们把前面学到的所有知识整合起来，搭建一个完整的 TypeScript 工程化体系——代码规范、类型安全、测试保障、monorepo 管理、npm 包发布。

一个项目"能跑"和"工程化"之间的差距，往往决定了团队效率的上限。没有 ESLint，代码风格各自为战；没有测试，重构如履薄冰；没有 strict 模式，类型形同虚设。本章解决的就是这些问题——让你的项目从"能编译"升级为"可维护"。

## ESLint 配置

### 安装

```bash
npm install -D eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin
npm install -D eslint-config-prettier eslint-plugin-prettier
```

### 配置文件

```javascript name=".eslintrc.cjs"
module.exports = {
  root: true,
  parser: '@typescript-eslint/parser',
  parserOptions: {
    ecmaVersion: 2022,
    sourceType: 'module',
    project: './tsconfig.json',
  },
  plugins: ['@typescript-eslint'],
  extends: [
    'eslint:recommended',
    'plugin:@typescript-eslint/recommended',
    'plugin:@typescript-eslint/recommended-requiring-type-checking',
    'prettier',
  ],
  rules: {
    // 类型相关
    '@typescript-eslint/no-explicit-any': 'error',
    '@typescript-eslint/no-unsafe-assignment': 'error',
    '@typescript-eslint/no-unsafe-call': 'error',
    '@typescript-eslint/no-unsafe-return': 'error',
    '@typescript-eslint/explicit-function-return-type': ['warn', {
      allowExpressions: true,
      allowTypedFunctionExpressions: true,
    }],

    // 代码质量
    '@typescript-eslint/no-unused-vars': ['error', { argsIgnorePattern: '^_' }],
    '@typescript-eslint/consistent-type-imports': ['error', {
      prefer: 'type-imports',
    }],
    '@typescript-eslint/no-floating-promises': 'error',
    '@typescript-eslint/no-misused-promises': 'error',

    // 命名规范
    '@typescript-eslint/naming-convention': ['error', {
      selector: 'interface',
      format: ['PascalCase'],
      custom: { regex: '^I[A-Z]', match: true },
    }, {
      selector: 'typeAlias',
      format: ['PascalCase'],
    }],
  },
}
```

### 常用规则说明

| 规则 | 作用 |
|------|------|
| `no-explicit-any` | 禁止使用 `any` |
| `no-unsafe-*` | 禁止不安全的类型操作 |
| `no-floating-promises` | Promise 必须被 await 或 catch |
| `no-misused-promises` | 禁止把 Promise 传给不需要的地方 |
| `consistent-type-imports` | 统一用 `import type` 导入类型 |

## 单元测试

为什么需要测试？因为**类型检查只能保证编译时正确，不能保证运行时行为**。`add(1, 2)` 的返回类型是 `number`，但实现可能是 `return 0`——类型检查完全通过，逻辑却完全错误。

### 为什么选 Vitest？

Vitest 是 Vite 团队出品的测试框架，相比 Jest 有两大优势：

1. **原生 ESM 支持**——不需要 Babel 转译，和 tsconfig 的 module 配置完全一致
2. **极快的启动速度**——基于 Vite 的即时编译，冷启动比 Jest 快 10 倍以上

### 安装与配置

```bash
npm install -D vitest
```

在 `package.json` 中添加脚本：

```json
{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest",
    "test:coverage": "vitest run --coverage"
  }
}
```

创建 `vitest.config.ts`：

```typescript name="vitest.config.ts"
import { defineConfig } from 'vitest/config'

export default defineConfig({
  test: {
    globals: true,        // 全局 API（describe/it/expect），不用每次 import
    environment: 'node',  // Node.js 环境
    include: ['src/**/*.test.ts'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
      include: ['src/**/*.ts'],
      exclude: ['src/**/*.test.ts', 'src/types/**'],
    },
  },
})
```

### 编写测试

以第 5 章的 TaskManager 为例：

```typescript name="src/core/TaskManager.test.ts"
import { describe, it, expect, beforeEach } from 'vitest'
import { TaskManager } from './TaskManager.js'

describe('TaskManager', () => {
  let manager: TaskManager

  beforeEach(() => {
    manager = new TaskManager()
  })

  describe('add', () => {
    it('应该创建任务并返回', () => {
      const task = manager.add('学习测试', 'high')
      expect(task.title).toBe('学习测试')
      expect(task.priority).toBe('high')
      expect(task.status).toBe('todo')
    })

    it('默认优先级应为 medium', () => {
      const task = manager.add('测试')
      expect(task.priority).toBe('medium')
    })
  })

  describe('update', () => {
    it('应该更新任务状态', () => {
      const task = manager.add('测试')
      const updated = manager.update(task.id, { status: 'done' })
      expect(updated?.status).toBe('done')
    })

    it('更新不存在的任务应返回 undefined', () => {
      const result = manager.update(999, { status: 'done' })
      expect(result).toBeUndefined()
    })
  })

  describe('list', () => {
    it('应该按筛选条件过滤', () => {
      manager.add('任务1', 'high')
      manager.add('任务2', 'low')

      const urgent = manager.list({ priority: 'high' })
      expect(urgent).toHaveLength(1)
      expect(urgent[0].priority).toBe('high')
    })
  })
})
```

### 测试类型安全

测试不仅是验证行为，也是验证类型的好机会。如果类型定义有误，测试代码本身就无法编译：

```typescript
// 如果 TaskManager.remove 的返回类型写错了（比如写成 void），
// 下面这行测试在编译时就会报错
it('删除存在的任务应返回 true', () => {
  manager.add('测试')
  const result = manager.remove(1)
  expect(result).toBe(true)  // 如果 remove 返回 void，这里编译不通过
})
```

## strict 模式实践

很多教程直接告诉你"开启 strict"，但没人解释**为什么**。核心原因是：**宽松模式下，TypeScript 给你的安全感是虚假的**。

```typescript
// strictNullChecks 关闭时，这段代码编译通过
function getLength(text: string | null): number {
  return text.length  // 运行时：如果 text 是 null，直接崩溃
}
```

你以为 TS 帮你检查了，实际上它放过了最常见的一类 bug——`null`/`undefined` 访问。strict 模式堵的就是这种漏洞。

`strict: true` 开启所有严格检查，这是 TypeScript 项目的**底线配置**：

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

等价于同时开启：

| 选项 | 作用 |
|------|------|
| `strictNullChecks` | null/undefined 不能赋给其他类型 |
| `noImplicitAny` | 禁止隐式 any |
| `strictFunctionTypes` | 函数参数类型协变检查 |
| `strictBindCallApply` | bind/call/apply 参数检查 |
| `strictPropertyInitialization` | 类属性必须初始化 |
| `noImplicitThis` | 禁止隐式 any 的 this |
| `useUnknownInCatchVariables` | catch 的 error 类型为 unknown |

### 常见 strict 模式问题

**null 检查**

```typescript
// ❌ strictNullChecks 下报错
function getLength(text: string | null): number {
  return text.length  // Object is possibly 'null'
}

// ✅ 方式一：类型守卫
function getLength(text: string | null): number {
  if (text === null) return 0
  return text.length
}

// ✅ 方式二：可选链
function getLength(text: string | null): number {
  return text?.length ?? 0
}
```

**类属性初始化**

```typescript
// ❌ strictPropertyInitialization 下报错
class User {
  name: string  // Property 'name' has no initializer
}

// ✅ 方式一：构造函数赋值
class User {
  name: string
  constructor(name: string) {
    this.name = name
  }
}

// ✅ 方式二：明确赋值断言
class User {
  name!: string  // 告诉 TS 我会在其他地方初始化
}
```

**catch 错误类型**

```typescript
try {
  // ...
} catch (error) {
  // error 类型为 unknown（不是 any）
  // ❌ error.message
  // ✅
  if (error instanceof Error) {
    console.log(error.message)
  }
}
```

## monorepo 管理

用 npm workspaces 管理多个包：

### 项目结构

```
task-manager/
├── packages/
│   ├── core/          # 核心类型和逻辑
│   │   ├── src/
│   │   ├── package.json
│   │   └── tsconfig.json
│   ├── storage/       # 存储适配器
│   │   ├── src/
│   │   ├── package.json
│   │   └── tsconfig.json
│   └── cli/           # 命令行界面
│       ├── src/
│       ├── package.json
│       └── tsconfig.json
├── package.json       # 根 package.json
└── tsconfig.base.json # 共享 TS 配置
```

### 根配置

```json name="package.json"
{
  "name": "task-manager",
  "private": true,
  "workspaces": ["packages/*"]
}
```

```json name="tsconfig.base.json"
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}
```

### 子包配置

```json name="packages/core/tsconfig.json"
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}
```

```json name="packages/core/package.json"
{
  "name": "@task-manager/core",
  "version": "1.0.0",
  "type": "module",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js"
    }
  },
  "scripts": {
    "build": "tsc",
    "dev": "tsc --watch",
    "typecheck": "tsc --noEmit"
  }
}
```

### 子包引用

```json name="packages/storage/package.json"
{
  "name": "@task-manager/storage",
  "dependencies": {
    "@task-manager/core": "workspace:*"
```

## npm 包发布

### 准备

1. 确保所有导出都有类型声明
2. 正确配置 `exports` 字段
3. 在 `.npmignore` 中排除源码

### package.json 关键字段

```json
{
  "name": "@your-scope/package-name",
  "version": "1.0.0",
  "description": "简短描述",
  "keywords": ["typescript", "task-manager"],
  "license": "MIT",
  "repository": { "type": "git", "url": "https://github.com/you/repo" },
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "files": ["dist"],
  "sideEffects": false,
  "peerDependencies": {},
  "devDependencies": {}
}
```
